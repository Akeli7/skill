# fix_bug

端到端修 bug 的链路式 skill。核心思想：用户给的往往是一句不完整的自然语言（缺陷单、
群里一句话、复现步骤缺失），直接去翻代码很容易跑偏。所以第一步先把它「补全 + 扩充查询」，
再沿链路推进，一次性完成从定位到修复的全流程。

链路：**信息补全 → 根因定位 → 修复设计 → 最小改动 → 审查收口**

---

## PHASE 0 — 信息补全与检索

目标：把用户的自然语言变成一份**信息齐全、带证据索引**的缺陷简报，并生成可执行的检索查询。
这一步不做代码改动，只做信息补全与检索。

### 0.1 补全缺陷简报

从用户原始输入里抽取，缺的字段就显式标「待确认」并问用户，不要脑补：

| 字段 | 说明 |
|------|------|
| 平台 / 端 | harmony / android / ios / 多端 / 后端，能定的就定 |
| 受影响模块 | 从输入里能猜到的就写，猜不到就靠检索反推 |
| 复现步骤 | 用户给的原文；缺失则列为第一待确认项 |
| 实际现象 | 用户描述的具体错误表现 |
| 预期行为 | 用户认为「应该怎样」 |
| 失败信号 | 报错栈 / UI 错乱 / 卡顿 / 逻辑反了 —— 这是后续建反馈环的命门 |

产出一份「缺陷简报」，字段齐全的打勾，缺失的标 `?`。

### 0.2 扩充查询

基于简报，生成三类检索查询，逐类执行并把结果附到简报：

1. **代码检索**：用 Grep / Glob / 读文件，按嫌疑符号、模块名、相关 feature 目录搜索。
   首项应是「报错里的关键符号」或「用户点名的功能」。
2. **文档检索**：读仓库里的 `CONTEXT.md`、ADR、相关 `docs/`（若该仓库有），拿架构心智模型。
3. **跨端参考**：同一功能在别的端怎么实现？搜对应端目录（如 android/、ios/），
   这是定位「不该这么走」的关键参照。

检索完，把命中的关键文件、行号、相关代码片段收进简报的「证据索引」。

### 0.3 提问规则

只有「复现步骤」和「预期行为」缺失到无法推进时才问用户；其余字段能检索推断的就推断。
一次最多问 2 个问题，且必须给出「我这边的默认假设」，让用户改而不是从零答。

**PHASE 0 完成判据**：简报字段齐（或已标 ? 并问过用户）、证据索引至少命中 1 个嫌疑文件、
跨端参考已查（或确认无对应端）。之后才进入 PHASE 1。

## PHASE 1 — 根因定位

基于缺陷简报和证据索引，建立反馈环并定位根因。

### 1.1 建立反馈环

**这是整个诊断的核心。** 如果你有一个**紧致**的 pass/fail 信号能触发这个 bug——
一个能在 bug 时变红、修复后变绿的信号——你就能找到根因。反馈环、假设检验、
打点验证都只是在消费这个信号。没有它，看再多代码也救不了你。

**构建反馈环的方式**（按优先级尝试）：

1. **失败测试**：在能触达 bug 的边界写单元/集成/E2E 测试
2. **HTTP 脚本**：对运行中的 dev server 发 curl
3. **CLI 调用**：用 fixture 输入调用，diff stdout 与预期快照
4. **无头浏览器脚本**：用 Playwright / Puppeteer 驱动 UI，断言 DOM/console/network
5. **回放捕获的 trace**：把真实的网络请求/载荷/事件日志保存到磁盘，隔离回放
6. **临时 harness**：启动系统最小子集（一个服务 + mock 依赖），单次函数调用触达 bug 路径
7. **属性/模糊循环**：如果 bug 是「有时输出错误」，跑 1000 随机输入找失败模式
8. **二分 harness**：如果 bug 出现在两个已知状态之间，自动化「在状态 X 启动并检查」以便 `git bisect run`
9. **差异循环**：同一输入跑旧版本 vs 新版本（或两个配置），diff 输出
10. **HITL 脚本**：最后手段。如果必须人工点击，用脚本驱动人工操作，保持循环结构化

### 1.2 收紧反馈环

把反馈环当作产品来打磨。一旦有了一个环，**收紧**它：

- 能让它更快吗？（缓存初始化、跳过无关 setup、收窄测试范围）
- 能让信号更锐吗？（断言具体症状，而不是「没报错」）
- 能让它更确定吗？（固定时间、种子随机数、隔离文件系统、冻结网络）

一个 30 秒的 flaky 循环几乎没用；一个 2 秒的确定性循环才是调试超能力。

### 1.3 处理非确定性 bug

目标不是干净复现，而是**更高的复现率**。循环触发 100 次、并行化、加压、
收窄时间窗口、注入 sleep。1% flaky 的 bug 不可调试；不断提率直到可调试。

### 1.4 无法构建反馈环时

停下来，明确说明。列出尝试过的方法。向用户请求：
- (a) 复现环境的访问权限
- (b) 捕获的工件（HAR 文件、日志 dump、core dump、带时间戳的录屏）
- (c) 允许添加临时生产监控

**不要**在没有反馈环的情况下开始假设。

### 1.5 复现并最小化

跑反馈环，观察变红——bug 出现。

确认：
- [ ] 环产出的是**用户**描述的失败模式——不是附近碰巧出现的不同失败
- [ ] 失败在多次运行中可复现（或非确定性 bug 以足够高的率复现以便调试）
- [ ] 已捕获精确症状（错误信息、错误输出、慢响应时间）供后续阶段验证

**最小化**：一旦变红，把复现场景**缩减到仍能变红的最小场景**。逐个削减输入、调用者、
配置、数据、步骤——每次削减后重新运行环——只保留承载负载的要素。

为什么费这个劲：最小复现把假设空间在阶段 3 中缩小（更少的活动部件留待怀疑），
并成为阶段 5 的干净回归测试。

完成标准：**每个剩余要素都是 load-bearing**——移除任何一个都会让环变绿。

**完成判据**：在复现**并且**最小化之前不要继续。

### 1.6 提出假设

在测试任何假设之前生成 **3–5 条排序的假设**。单一假设生成会锚定在第一个看似合理的想法上。

每条假设必须**可证伪**：说明它预测的结果。

> 格式：「如果 <X> 是原因，那么 <改变 Y> 会让 bug 消失 / <改变 Z> 会让它更严重。」

如果无法说明预测，这个假设就是一个 vibe——丢掉或把它改精确。

**把排序列表展示给用户再测试。** 他们通常有领域知识能立即重新排序
（「我们刚部署了对 #3 的改动」），或知道已经排除的假设。便宜的检查点，省大时间。
不要因此阻塞——如果用户 AFK 就按你的排序继续。

### 1.7 打点验证

每个探针必须映射到阶段 1.6 中的具体预测。**一次只改变一个变量。**

工具偏好：
1. **调试器 / REPL 检查**：如果环境支持，一个断点胜过十条日志
2. **目标日志**：在区分假设的边界处加日志
3. **永远不要**「打所有日志然后 grep」

**给每条调试日志打唯一标签**，如 `[DEBUG-a4f2]`。最后清理变成一次 grep。
无标签日志存活；带标签日志删除。

### 1.8 修复 + 回归测试

在**正确的位置**先写回归测试——但前提是存在正确的 seam。

正确的 seam 是测试能行使**真实 bug 模式**的位置，就像它在调用点发生的那样。
如果唯一可用的 seam 太浅（单调用者测试当 bug 需要多个调用者，单元测试无法复制触发 bug 的链），
在那里写回归测试会给出虚假信心。

**如果不存在正确 seam，这本身就是发现。** 记录它。代码架构正在阻止 bug 被锁定。
把它交给下一阶段。

如果存在正确 seam：
1. 把最小化复现转成该 seam 处的失败测试
2. 观察它失败
3. 应用修复
4. 观察它通过
5. 用原始（未最小化）场景重新运行阶段 1 反馈环

### 1.9 清理 + 复盘

在宣布完成之前必须完成：

- [ ] 原始复现不再复现（重新运行阶段 1 的环）
- [ ] 回归测试通过（或不存在 seam 已记录）
- [ ] 所有 `[DEBUG-...]` 监控已删除（grep 标签）
- [ ] 临时原型已删除（或移到明确标记的调试位置）
- [ ] 被证实正确的假设写在 commit / PR message 中——这样下一个调试者能学到

**然后问：什么能预防这个 bug？** 如果答案涉及架构变更
（没有好的测试 seam、调用者纠缠、隐藏耦合），在修复完成后交给下一阶段，
不是修复前——现在你比开始时有更多信息。

**完成判据**：能明确指出「什么条件下、哪段代码路径」导致了缺陷。

## PHASE 2 — 修复设计

基于根因，确定「修复该落在哪个位置」。

### 2.1 设计原则：深模块

设计**深模块**：大量行为藏在小接口后面，放在干净的 seam 处，
通过该接口可测试。目标是给调用者杠杆、给维护者局部性、给所有人可测试性。

**核心概念**：

| 概念 | 定义 |
|------|------|
| **Module** | 有接口和实现的任何东西。故意不区分规模：函数、类、包或跨层切片 |
| **Interface** | 调用者正确使用模块所需知道的一切：类型签名、不变量、顺序约束、错误模式、必需配置、性能特征 |
| **Implementation** | 模块内部的东西，它的代码主体 |
| **Depth** | 接口处的杠杆：调用者（或测试）能行使的行为量 / 需要学习的接口单位 |
| **Seam** | Michael Feathers 的定义：你可以在不编辑该位置的情况下改变行为的位置 |
| **Adapter** | 满足 seam 处接口的具体实现 |
| **Leverage** | 调用者从深度获得的：每学习一个接口单位获得更多能力 |
| **Locality** | 维护者从深度获得的：变更、bug、知识、验证集中在一个地方 |

**深模块** = 小接口 + 大量实现：

```
┌─────────────────────┐
│   Small Interface   │  ← 少方法，简单参数
├─────────────────────┤
│                     │
│  Deep Implementation│  ← 复杂逻辑隐藏
│                     │
└─────────────────────┘
```

**浅模块** = 大接口 + 小实现（应避免）：

```
┌─────────────────────────────────┐
│       Large Interface           │  ← 多方法，复杂参数
├─────────────────────────────────┤
│  Thin Implementation            │  ← 只是透传
└─────────────────────────────────┘
```

### 2.2 设计原则

- **深度是接口的属性，不是实现的。** 深模块的内部可以由小的、可 mock 的、可替换的部分组成——
  它们只是不在接口中。模块可以有**内部 seam**（私有于实现，供自身测试）和
  **外部 seam**（在接口处）。

- **删除测试。** 想象删除这个模块。如果复杂度消失，它就是透传。如果复杂度
  重新出现在 N 个调用者中，它就物有所值。

- **接口是测试表面。** 调用者和测试穿越同一个 seam。如果想测试接口**之后**的东西，
  模块的形状可能不对。

- **一个 adapter = 假设性的 seam。两个 adapter = 真正的 seam。** 不要在没有东西
  在其上变化时引入 seam。

### 2.3 设计模式

修复设计应遵循常用设计模式，满足 SOLID 原则：

- **单一职责原则 (SRP)**：每个类/模块只负责一项工作
- **开闭原则 (OCP)**：对扩展开放，对修改关闭
- **里氏替换原则 (LSP)**：子类型必须能替换其基类型
- **接口隔离原则 (ISP)**：客户端不应依赖它不使用的接口
- **依赖倒置原则 (DIP)**：高层模块不应依赖低层模块，都应依赖抽象

常用设计模式参考：
- **策略模式**：当不同算法需要在运行时切换时
- **工厂模式**：当对象创建逻辑复杂或需要延迟决策时
- **观察者模式**：当需要通知多个订阅者状态变化时
- **装饰器模式**：当需要给对象添加职责而不修改其接口时
- **模板方法模式**：当有固定算法骨架但步骤实现可变时

### 2.4 设计可测试性

好的接口让测试自然：

1. **接受依赖，不要创建它们。**
   ```typescript
   // 可测试
   function processOrder(order, paymentGateway) {}
   // 难以测试
   function processOrder(order) {
     const gateway = new StripeGateway();
   }
   ```

2. **返回结果，不要产生副作用。**
   ```typescript
   // 可测试
   function calculateDiscount(cart): Discount {}
   // 难以测试
   function applyDiscount(cart): void {
     cart.total -= discount;
   }
   ```

3. **小表面面积。** 更少的方法 = 更少的测试。更少的参数 = 更简单的测试 setup。

### 2.5 修复设计要点

- 优先用**方法覆写（override hook）**而不是往基类/接口塞新字段/新参数。
  新字段只在「它真正被多处复用、且删掉它复杂度会扩散到 N 个调用点」时才引入（deletion test）。
- 接口要小：用基类暴露一个 `isXxxDisabled()` 之类的钩子，子类覆写，调用方无感。
- 确认改动是「深模块」式的：行为藏在小接口后面，调用方不必知道内部状态。

**完成判据**：明确修复落点（哪个类/方法、是新增 hook 还是覆写、是否需要新字段）+ 理由。

## PHASE 3 — 最小改动落地

按修复设计落地代码改动，遵循「懒人」原则：用最少的代码解决实际问题。

### 3.1 改动阶梯

从最高优先级开始，停在第一个能用的：

1. **这需要存在吗？** 投机性需求 = 跳过它，一句话说明。YAGNI
2. **标准库能做吗？** 用它
3. **原生平台特性覆盖吗？** `<input type="date">` 胜过 picker 库，CSS 胜过 JS，DB 约束胜过应用代码
4. **已安装依赖能解决吗？** 用它。不要为几行代码能做的事加新依赖
5. **能一行搞定吗？** 一行
6. **只有这样：** 能用的最小代码

第一个能用的懒方案就是正确的方案。

### 3.2 落地规则

- 不要未请求的抽象：没有一个实现的接口、一个产品一个工厂、永远不变的值的配置
- 不要样板代码、不要「以后会用到」的脚手架——以后可以自己搭
- 删除优于添加。无聊优于聪明，聪明是别人凌晨 3 点要解码的东西
- 尽可能少的文件。最短能用的 diff 获胜
- 复杂请求？先交付懒版本并质疑，「做了 X；Y 能覆盖它。需要完整 X？说一声。」
  不要因等待答案而停滞
- 两个标准库选项大小相同？选边角用例正确的那个。懒意味着写更少代码，不是选更弱的算法
- 标记有意的简化用 `ponytail:` 注释（`// ponytail: 这个存在`），简单被读作意图，
  不是无知。有已知上限的捷径（全局锁、O(n²) 扫描、朴素启发式）？注释命名上限和升级路径：
  `# ponytail: 全局锁，按吞吐量需要时升级为按账户锁`

### 3.3 什么时候不该懒

永远不要简化掉：信任边界处的输入验证、防止数据丢失的错误处理、
安全措施、无障碍基础、用户明确请求的任何东西。
用户坚持完整版本 → 构建它，不要重新争论。

硬件从来不是纸面上的理想：真实时钟有漂移、真实传感器读数有偏差、
PCA9685 跑得快几个百分点。留下校准旋钮，不只是更少的代码——物理世界
需要最小模型看不到的调优。

懒代码没有检查就是未完成。非平凡逻辑（分支、循环、解析器、资金/安全路径）
留下一个可运行的检查，如果逻辑坏了会失败：
一个基于 `assert` 的 `demo()`/`__main__` 自检或一个小的 `test_*.py`。
不要框架、不要 fixtures、不要每个函数的套件除非要求。平凡一行不需要测试，
YAGNI 对测试也适用。

### 3.4 输出原则

代码优先。然后最多三行短文：跳了什么、什么时候加回来。
不要长文、不要功能介绍、不要设计笔记。如果解释比代码长，
删掉解释——每段为简化而写的辩护都是伪装成散文的复杂性。

模式：`[代码] → 跳过：[X]，当 [Y] 时加回。`

**完成判据**：实际代码改动是最小集，无冗余抽象。

## PHASE 4 — 审查收口

对本次改动做双轴审查。

### 4.1 Standards 轴

代码是否符合本仓库的编码规范？

检查来源（按优先级）：
1. commit message 中的 issue 引用（`#123`、`Closes #45`、GitLab `!67` 等）
   — 通过 issue tracker 获取
2. 用户传入的路径
3. `docs/`、`specs/` 或 `.scratch/` 下匹配分支名或功能的 PRD/spec 文件
4. 如果找不到，问用户 spec 在哪。如果用户说没有，**Spec** 轴跳过并报告「无可用 spec」

### 4.2 Spec 轴

代码是否忠实实现了原始 issue / PRD / spec？

检查项：
- spec 要求但缺失或部分实现的需求
- diff 中未被要求的行为（范围蔓延）
- 看起来实现了但实现可能有误的需求

### 4.3 并行审查

两个轴**并行**作为子 agent 运行，这样它们不会污染对方的上下文，
然后本 skill 汇总发现。

每个子 agent 的报告：
- 每个违反规范的地方，按文件/hunk 标注相关规范
- 区分硬性违规和判断调用
- 跳过工具强制执行的
- 400 字以内

### 4.4 汇总

在 `## Standards` 和 `## Spec` 标题下列出两个报告，逐字或轻度清理。
**不要**合并或重新排序发现——两个轴是有意分开的。

以一行总结结束：每轴的发现总数，以及每个轴的最严重问题（如果有）。
不要跨轴选单一赢家——那是分开存在的原因。

### 4.5 为什么是两个轴

一个改动可能通过一个轴而另一个轴失败：
- 代码遵循每个规范但实现了错误的东西 → **Standards 通过，Spec 失败**
- 代码精确实现 issue 要求但违反项目约定 → **Spec 通过，Standards 失败**

分开报告防止一个轴掩盖另一个轴。

**完成判据**：反馈环转绿 + 审查无遗留 + 改动是最小集。

## 为什么是这条链

- 信息补全解决「输入太碎、易跑偏」——先把问题讲清楚再动刀。
- 根因定位保证「有证据、不靠猜」。
- 修复设计保证「改对地方」，避免补丁叠补丁、往基类乱塞字段。
- 最小改动保证「改得最小」，杜绝过度工程。
- 审查收口保证「改得对、改得合规」，闭环结束。
