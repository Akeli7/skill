# fix_bug

端到端修 bug 的链路式 skill。核心思想：用户给的往往是一句不完整的自然语言（缺陷单、
群里一句话、复现步骤缺失），直接去翻代码很容易跑偏。所以第一步先把它「补全 + 扩充查询」，
再沿四段链路推进，每段由对应子 skill 接管。

链路：**RAG 扩充补全 → diagnosing-bugs → codebase-design → ponytail → review**

每一段都是前一段的输入；前一段没有产出，就不要在它之前开工。

> 四段子 skill 均通过 Skill 工具加载：在对应阶段调用 Skill 工具并传入
> `diagnosing-bugs` / `codebase-design` / `ponytail` / `review` 即可。

---

## PHASE 0 — RAG 扩充 / 补全 / 增加查询（本 skill 负责）

目标：把用户的自然语言变成一份**信息齐全、带证据索引**的缺陷简报，并生成可执行的检索查询。
这一步不做代码改动，只做信息补全与检索。

### 0.1 补全缺陷简报（补全）

从用户原始输入里抽取，缺的字段就显式标「待确认」并问用户（见下方提问规则），不要脑补：

| 字段 | 说明 |
|------|------|
| 平台 / 端 | harmony / android / ios / 多端 / 后端，能定的就定 |
| 受影响模块 | 从输入里能猜到的就写，猜不到就靠 0.2 的检索反推 |
| 复现步骤 | 用户给的原文；缺失则列为第一待确认项 |
| 实际现象 | 用户描述的具体错误表现 |
| 预期行为 | 用户认为「应该怎样」 |
| 失败信号 | 报错栈 / UI 错乱 / 卡顿 / 逻辑反了 —— 这是后续建反馈环的命门 |

产出一份「缺陷简报」，字段齐全的打勾，缺失的标 `?`。

### 0.2 增加查询（扩充）

基于简报，生成三类检索查询，逐类执行并把结果附到简报：

1. **代码检索**：用 Grep / Glob / 读文件，按嫌疑符号、模块名、相关 feature 目录搜索。
   首项应是「报错里的关键符号」或「用户点名的功能」。
2. **文档检索**：读仓库里的 `CONTEXT.md`、ADR、相关 `docs/`（若该仓库有），拿架构心智模型。
3. **跨端参考**：同一功能在别的端怎么实现？搜对应端目录（如 android/、ios/），
   这正是定位「不该这么走」的关键参照。

检索完，把命中的关键文件、行号、相关代码片段收进简报的「证据索引」。

### 0.3 提问规则

只有「复现步骤」和「预期行为」缺失到无法推进时才问用户；其余字段能检索推断的就推断。
一次最多问 2 个问题，且必须给出「我这边的默认假设」，让用户改而不是从零答。

**PHASE 0 完成判据**：简报字段齐（或已标 ? 并问过用户）、证据索引至少命中 1 个嫌疑文件、
跨端参考已查（或确认无对应端）。之后才进入 PHASE 1。

## PHASE 1 — diagnosing-bugs

用 Skill 工具加载 `diagnosing-bugs`，把 PHASE 0 的缺陷简报 + 证据索引作为它的输入。

按它的六阶段推进：**建反馈环 → 复现并最小化 → 提假设 → 打点验证 → 修 + 回归测试 → 清理**。

重点：
- 反馈环要对准 PHASE 0 记录的「失败信号」，必须能 red、能 green。
- 假设要 3–5 条并排序，先给用户看一眼。
- 打完点后，把「真正的根因」回写进缺陷简报，作为 PHASE 2 的输入。

**交接产物**：根因定位（含命中的代码路径与触发条件）。

---

## PHASE 2 — codebase-design

用 Skill 工具加载 `codebase-design`，基于根因找「修复该落在哪个 seam」。

要点（与本 skill 强相关）：
- 优先用**方法覆写（override hook）**而不是往基类/接口塞新字段/新参数。
  新字段只在「它真正被多处复用、且删掉它复杂度会扩散到 N 个调用点」时才引入（deletion test）。
- 接口要小：用基类暴露一个 `isXxxDisabled()` 之类的钩子，子类覆写，调用方无感。
- 确认改动是「深模块」式的：行为藏在小接口后面，调用方不必知道内部状态。

**交接产物**：修复落点（哪个类/方法、是新增 hook 还是覆写、是否需要新字段）+ 理由。

---

## PHASE 3 — ponytail

用 Skill 工具加载 `ponytail`（默认 full），按它的 ladder 落地 PHASE 2 选的修复。

要点：
- 第一反应问「这改动非加不可吗」；能复用已有数据（已有 Map/状态）就别新增。
- 最短 diff 获胜；不写未请求的抽象、不堆注释。
- 落地后，把改动 diff 收好，交给 PHASE 4。

**交接产物**：实际代码改动（最小集）。

---

## PHASE 4 — review

用 Skill 工具加载 `review`，对本次分支从分支点（或用户给的 fixed point）做双轴审查：

- **Standards**：是否符合本仓库编码规范。
- **Spec**：是否真的修掉了 PHASE 0 简报里的缺陷、有无 scope creep。

若任一轴有发现，回到 PHASE 3 调整；干净则结束。

**完成判据**：PHASE 1 的反馈环转绿 + PHASE 4 审查无遗留 + 改动是最小集。

---

## 为什么是这条链

- RAG 补全解决「输入太碎、易跑偏」——先把问题讲清楚再动刀。
- diagnosing-bugs 保证「有证据、不靠猜」地定位根因。
- codebase-design 保证「改对地方」，避免补丁叠补丁、往基类乱塞字段。
- ponytail 保证「改得最小」，杜绝过度工程。
- review 保证「改得对、改得合规」，闭环收口。
