# fix_bug

端到端修 bug 的链路式 WorkBuddy skill。用户甩来一段碎碎念（缺陷单、群里一句话、复现步骤缺失），直接去翻代码很容易跑偏。这个 skill 的做法是先把它补全，再沿链路推进，一次性完成从定位到修复的全流程。

## 链路

```
信息补全 → 根因定位 → 修复设计 → 最小改动 → 审查收口
```

## 每段在干什么

**PHASE 0 — 信息补全与检索**

把用户的自然语言变成一份信息齐全、带证据索引的缺陷简报。这一步不做代码改动，只做信息补全与检索。

核心动作：
- 抽出平台、模块、复现步骤、实际现象、预期行为、失败信号
- 生成三类检索查询：代码检索、文档检索、跨端参考
- 缺「复现步骤」或「预期行为」时才问用户，其余能推断就推断

**PHASE 1 — 根因定位**

基于缺陷简报和证据索引，建立反馈环并定位根因。

核心动作：
1. **建反馈环**：针对「失败信号」设计可 red、可 green 的验证方式
2. **复现并最小化**：把复现路径缩短到最简
3. **提假设**：列出 3–5 条可能的根因，按可能性排序
4. **打点验证**：逐个验证假设，锁定真正的根因
5. **回写简报**：把根因定位结果回写进缺陷简报

反馈环构建方式（按优先级）：
- 失败测试 → HTTP 脚本 → CLI 调用 → 无头浏览器 → 回放 trace → 临时 harness → 属性/模糊循环 → 二分 harness → 差异循环 → HITL 脚本

**PHASE 2 — 修复设计**

基于根因，确定「修复该落在哪个位置」。

设计原则：
- **深模块**：大量行为藏在小接口后面
- **删除测试**：删除模块后复杂度是否重新出现在 N 个调用者中
- **接口是测试表面**：调用者和测试穿越同一个 seam

设计模式：
- 遵循 SOLID 原则（单一职责、开闭、里氏替换、接口隔离、依赖倒置）
- 常用模式：策略模式、工厂模式、观察者模式、装饰器模式、模板方法模式

**PHASE 3 — 最小改动落地**

按修复设计落地代码改动，遵循「懒人」原则。

改动阶梯（从高到低）：
1. 这需要存在吗？→ YAGNI，跳过
2. 标准库能做吗？→ 用它
3. 原生平台特性覆盖吗？→ 用它
4. 已安装依赖能解决吗？→ 用它
5. 能一行搞定吗？→ 一行
6. 最小能用的代码

**PHASE 4 — 审查收口**

双轴审查：
- **Standards**：是否符合本仓库编码规范
- **Spec**：是否真修掉了缺陷、有无范围蔓延

## 为什么是这条链

- 信息补全解决「输入太碎、易跑偏」——先把问题讲清楚再动刀
- 根因定位保证「有证据、不靠猜」
- 修复设计保证「改对地方」，避免补丁叠补丁
- 最小改动保证「改得最小」，杜绝过度工程
- 审查收口保证「改得对、改得合规」，闭环结束

## 安装

把整个 fix_bug 目录复制到 `~/.workbuddy/skills/fix_bug/`，重启 WorkBuddy 即可。

## 触发方式

在 WorkBuddy 对话中说「修 bug」「定位这个缺陷」「排查这个问题」或直接贴一段缺陷描述即可触发。
